# hersona-duet 計画書 v2 — エージェント協働の自動制作スタジオ

> 作成日: 2026-06-12 (v2) / ステータス: ドラフト（リポジトリ分離前の設計合意用）
> v1 (ギャルゲーエンジン構想) をオーナーの具体例に基づき改訂。
> **ギャルゲーは「コンテンツパックの一つ」に降格し、本体は「監督・シナリオライター・
> ナレーター・アクターの各エージェントが協働してシーン単位で作品を作るスタジオ」とする。**
> hersona は登場人物の人格ライブラリとして、ReYMeN Agent はエージェント実行基盤として活用する。

---

## 1. プロダクト定義（オーナー提示の具体例 = 要件）

### 1.1 ユーザーがやること

| 設定 | 内容 | hersona 利用 |
|---|---|---|
| 基本設定 | 世界観・どのような物語か | — |
| スタッフ設定 | 制作のプロとしてのプロンプトを投入（シナリオライター/ナレーター等） | **使わない**（職能プロンプト） |
| 登場人物設定 | キャラクターの人格・口調 | **使う**（属性ブレンド） |
| シーン設定 | 「買い物のシーン」「自宅のシーン」などシーン単位で区切って制作 | — |
| エージェント設定 | 各エージェントに使う LLM（プロバイダ/モデル）を個別指定 | — |

### 1.2 エージェント構成

```
監督 (= ユーザーの席)
 │  クオリティ管理・作品レビュー・リテイク指示。AI 補佐は任意の後付け
 ▼
シナリオライター (スタッフ / 職能プロンプト)
 │  シーンの起承転結（ビートシート）を提示。台詞は書かない＝演者に任せる
 ▼
ナレーター (スタッフ / 職能プロンプト)
 │  会話シーンの風景・情景描写を担当
 ▼
アクター ×N (hersona 属性ブレンドで人格注入)
    シナリオライターの進行とナレーターの描写に応じて、人格に従い発言
```

### 1.3 ユーザーの参加モード

1. **完全自動**: エージェントだけでシーンを作り上げる
2. **参加モード**: ユーザーが登場人物の一人としてチャットでシーンに入る。
   この場合も**ユーザー自身の登場人物設定（登場設定）は必須**
   （名前・役割は必須、hersona ブレンドは任意 — 「素の自分」で出てもよい）

人間の介入点は2層: **ターン単位**（参加モードでの発言）と**シーン単位**（監督レビュー →
採用 / 注記付きリテイク）。

### 1.4 UI（4 画面）

| 画面 | 内容 |
|---|---|
| 基本設定 | 世界観・物語・作品メタ |
| エージェント設定 | スタッフプロンプト編集・アクター配役（hersona 属性ピッカー）・各エージェントの LLM 指定 |
| シーン設定 | シーンの場所/目的/参加キャラ・シーン一覧（作品はシーンの束） |
| テキスト出力 | 完成シーンの台本体 / 小説体表示・Markdown エクスポート |

### 1.5 ReYMeN Agent との関係

- orchestrator（進行ロジック）はフレームワーク非依存の Python コアとして実装
- **ReYMeN Agent はエージェント実行アダプタの第一実装**（アクター/スタッフを ReYMeN
  エージェントとして駆動、`/duet` スキル提供）。スタンドアロン（API 直叩き）も併存
- hersona は配役層: 属性ブレンド → アクターの注入ブロック、相性マトリクス → 配役の
  ドラマ設計、intensity → 演技のキャラ一貫性 QA

## 2. 競合・近接事例（2026-06 調査）

### 2.1 制作スタジオ型（v2 の本体に対する近接領域）

| 事例 | 形態 | 評価 |
|---|---|---|
| **Co-DIRECT** (2025) | 学術 | **人間の監督を中心に置く Director-in-the-Loop + Writer/Actor/Critic エージェント構成**。本計画とほぼ同型の構造が学術的に有効と示されている。プロダクト/OSS ではない |
| **IBSEN** (2024) | 学術 | director エージェントが actor 群を統制する脚本生成。プロット追従と即興のバランスの先行知見 |
| Dramatron / StoryWriter / Agents' Room | 学術 | 階層分解・役割分担による長編生成。ビートシート方式の裏付け |
| **Fable Showrunner (SHOW-1)** | 商用 | マルチエージェントシミュレーションで TV エピソードを生成。クローズド・映像志向 |
| SillyTavern (グループチャット) | OSS アプリ | キャラ同士の雑談は可能だが、脚本・ビート進行・監督レビュー・シーン管理を持たない |
| VinA | OSS (停止) | プロンプト→ノベルゲーム一括生成の PoC。エージェント協働でも逐次制作でもない |

### 2.2 結論

> 「監督(人間)＋脚本＋ナレーター＋役者」構成は**学術的に有効性が確認された設計**だが、
> **OSS プロダクトとしての実装は存在しない**。さらに ①役者の人格を構造化属性
> （hersona）で定義 ②ユーザーが役として乱入できる ③シーン単位の制作管理 UI
> ④日英バイリンガル、の組み合わせは完全な空白。

差別化の核は v1 と同じ: 模倣コストが高いのは LLM 呼び出しではなく **hersona 側の決定的
データ資産**（64 属性・相性・weight・intensity）。スタジオが普及するほどライブラリの堀が
深くなる構造を維持する。

## 3. アーキテクチャ

### 3.1 リポジトリ構成（hersona-duet / 新リポジトリ / MIT）

```
hersona-duet/
├── duet/
│   ├── orchestrator.py      # シーン進行ループ (ビート駆動)
│   ├── agents/
│   │   ├── base.py          # Agent インターフェース (LLM 設定を個別に持つ)
│   │   ├── writer.py        # シナリオライター: シーン → ビートシート (起承転結)
│   │   ├── narrator.py      # ナレーター: ビート/直前の会話 → 情景描写
│   │   ├── actor.py         # アクター: hersona 注入 + 進行文脈 → 発話
│   │   └── human.py         # 人間アクター (参加モード): actor と同一インターフェース
│   ├── casting.py           # 配役: ブレンド解決・相性チェック (hersona API)
│   ├── review.py            # 監督レビュー: 採用 / 注記付きリテイク (シーン単位)
│   ├── providers.py         # LLM プロバイダ抽象 (anthropic / openai / ollama)
│   ├── script.py            # 台本データモデル + 出力 (台本体 / 小説体 Markdown)
│   ├── quality.py           # intensity 連携 (アクターのキャラ一貫性バッジ)
│   └── cli.py               # `duet scene run` / `duet scene join` / `duet export`
├── adapters/hermes/         # ReYMeN Agent 統合 (/duet スキル)
├── ui/                      # Phase 3: Web 4 画面 (ブラウザ完結・BYO キー)
├── packs/                   # プロジェクト保存形式: 世界観+スタッフ+配役+シーン束 (YAML)
│   └── examples/
└── docs/DESIGN.md
```

### 3.2 シーン進行ループ（中核）

```
シーン設定 (場所/目的/参加キャラ)
  → ①ライター: ビートシート生成 (起/承/転/結 各 1-3 ビート、台詞なし)
  → ②各ビート:
       ナレーター: 情景・状況の描写 (1-3 文)
       アクター群: ターンテイキングで発話
         - 発話者選択: ビートの指定 > 直前発話での言及 > ラウンドロビン
         - 参加モード時は human.py が割込み可能 (チャット入力)
       ビート消化判定 (ライターが「このビートの目的を満たしたか」を判定)
  → ③シーン完成 → 監督 (ユーザー) レビュー
       採用 → script へ確定 / リテイク → 注記を付けて ② を部分再生成
  → ④テキスト出力 (台本体 / 小説体)
```

- 「台詞はライターが書かず演者に任せる」分業が、よくある**全文一括生成の平板さ**を
  避ける本プロダクトの中心仮説（Co-DIRECT / IBSEN の知見とも一致）
- 同意ループ対策: ビートに「対立の種」フィールドを持たせ、hersona の conflicts
  データを燃料として供給する

### 3.3 エージェントごとの LLM 設定

```yaml
# packs/example/agents.yaml
writer:   { provider: anthropic, model: claude-opus-x,  prompt: prompts/writer_pro.md }
narrator: { provider: anthropic, model: claude-haiku-x, prompt: prompts/narrator.md }
actors:
  - name: 凛花
    hersona: [tsundere, kyoto_ben]
    weight: moderate
    llm: { provider: anthropic, model: claude-haiku-x }
  - name: あなた          # 参加モード: 登場設定は必須
    human: true
    role: 幼なじみ
    hersona: []           # 任意 (素の自分で参加可)
```

## 4. hersona 側の前提タスク（Phase 0 — v1 から変更なし）

| # | タスク | 規模 |
|---|---|---|
| P0-1 | 配布手段の確定（PyPI 公開 推奨 / 当面 git タグ依存可） | 2-4h |
| P0-2 | 公開 API の明文化（blend/attach・compatibility・weight・intensity・resolve_content_field を semver 対象と宣言） | 2h |

## 5. フェーズ計画（週 5-10h 前提）

> **実装エージェント向けの詳細設計は [`docs/duet/`](./duet/README.md) に分冊**
> （Phase 0〜4 の各文書に、モジュール仕様・データモデル・API 契約・テスト計画・
> 受け入れ基準を実装着手可能な粒度で記載）。本節は概要のみ。

### Phase 1: orchestrator CLI — 完全自動で 1 シーン（4-5 週）

- [ ] agents/base + providers: エージェント個別の LLM 設定（YAML）
- [ ] writer: シーン設定 → ビートシート（起承転結、台詞なし）
- [ ] narrator / actor: ビート駆動の描写と発話（actor は hersona 注入）
- [ ] orchestrator: ターンテイキング・ビート消化判定
- [ ] script: 台本体 / 小説体 Markdown 出力
- [ ] quality: `--measure` でアクターごとの intensity バッジ
- 完了基準: `duet scene run shopping.yaml` で「買い物のシーン」が監督レビュー可能な
  品質で出力される

### Phase 2: 監督ループ + 参加モード（3-4 週）

- [ ] review: シーン単位の採用 / 注記付きリテイク（注記がライターと演者に伝播）
- [ ] human actor: チャットでシーンに入る参加モード（登場設定の必須バリデーション）
- [ ] packs: 基本設定・スタッフ・配役・シーン束のプロジェクト保存形式
- 完了基準: ユーザーが幼なじみ役で乱入した買い物シーンを、リテイク 1 回を挟んで完成できる

### Phase 3: Web UI 4 画面（5-7 週）

- [ ] 基本設定 / エージェント設定 / シーン設定 / テキスト出力
- [ ] ブラウザ完結（BYO API キーをローカル保存・GitHub Pages 配信でホスト費ゼロ）
- [ ] 参加モードのチャット UI（シーン進行に発言を差し込む）

### Phase 4: 生態系（以降随時）

- [ ] ReYMeN Agent アダプタ（`/duet` スキル）
- [ ] ギャルゲーパック: P0-3 写像で「好感度→weight」を使う恋愛シーン集（v1 構想の回収）
- [ ] Character Card V2 インポータ（SillyTavern 圏からの配役流入）
- [ ] duet の完成シーンを X グロース施策の定常コンテンツに

## 6. リスク（正直版）

| リスク | 影響 | 対策 |
|---|---|---|
| マルチエージェント分の API コスト増（1 シーン = writer 1 + narrator×ビート + actor×ターン ≒ 数十〜百数十円/シーン） | 普及の摩擦 | エージェント別 LLM 指定が効く設計（演者は安価モデル、ライターのみ高級モデル）。ollama アダプタを Phase 1 から |
| ビート消化判定の精度 | 進行が間延び / 暴走 | ターン予算の強制上限 + 監督リテイクで人間が止められる構造 |
| スタッフプロンプトの質が作品品質のボトルネック | 「使ったけど微妙」評価 | プロ品質のプロンプトテンプレ（writer/narrator）を同梱し、差し替え可能に |
| UI 開発の重さ（週 5-10h で 3 フェーズ） | 停滞 | CLI 先行で価値検証 → UI は Phase 3 まで遅延。4 画面以上に増やさない |
| 同意ループ / キャラ崩壊 | 作品が平板に | ビートの「対立の種」+ conflicts データ + intensity バッジ |
| 18 禁ドリフト・固有名詞流入 | 規約・権利 | 全年齢の明文化 / パック検証に固有名詞ガード（hersona と同方針） |

## 7. 意思決定の記録

- **v2 改訂**: 本体は「ギャルゲーエンジン」ではなく「シーン単位の自動制作スタジオ」。
  ギャルゲーは Phase 4 のコンテンツパックとして回収（オーナー指示 2026-06-12）
- 監督は**ユーザーの席**（クオリティ管理・レビュー・リテイク）。AI 監督補佐は任意の後付け
- スタッフ（ライター/ナレーター）は hersona を使わず職能プロンプト、
  アクターのみ hersona を使う——この分離を API/データモデルにも反映する
- リポジトリは hersona と分離（`hersona-duet`）。hersona はライブラリとして
  `hersona>=1.2,<2` で依存。core は LLM 非依存・決定的のまま
- ユーザー参加時も登場設定は必須（名前・役割必須、hersona ブレンド任意）
