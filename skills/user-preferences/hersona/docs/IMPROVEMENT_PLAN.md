# hersona グロース改善計画書（2026-06 〜 2027-06）

> 作成日: 2026-06-11 / 対象: shiro-0x/hersona
> 前提: 個人開発・週5-10時間・予算ほぼゼロ / コアターゲット = 日本のAI×キャラ文化層
> 目的: ①OSS普及 ②X広報AIエージェント運用 ③収益化（bags系トークン発行を検討）

---

## 0. エグゼクティブサマリ

**診断結論: hersonaの制約（ボトルネック）は製品ではなく「認知」である。**
リポジトリは作成から1週間（2026-06-04）で Star 0 / Fork 0 / 外部コントリビューター 0。一方で
製品側は 64属性・相性マトリクス・診断クイズ・強度測定・i18n・テスト180件と、個人OSSとして
異例に成熟している。**投下労力の推定9割が製品、1割未満が流通に向いており、Pareto的に逆転が必要。**

**戦略の核は3つ:**

1. **「Xエージェント自身が hersona の最大のデモ」構造を作る** — 広報botに hersona の属性で
   人格を注入し、毎週フォロワー投票で人格を入れ替える。広報・デモ・エンゲージメントを1つの
   施策で兼ねる（後述「今週の人格総選挙」）。
2. **シェアされる単位を「リポジトリ」から「診断結果」に変える** — 既存のWeb診断クイズに
   結果シェアカード（OGP画像）を付け、「あなたのAIはこんな性格」をXに流すバイラルループを作る。
3. **収益化は順序が命** — トークンはコミュニティ不在のまま発行しても流動性ゼロで終わるだけでなく、
   日本のキャラ文化層の信頼を恒久的に毀損するリスクが最大。**GitHub Sponsors → 発行ゲート条件
   充足後にトークン再判断**の2段構えとする（§3.4 / 付録C）。

**正直な指摘（重要）:**

- 希望された「返信まで完全自動のXエージェント」は、メンション読み取りに X API 有料プラン
  （Basic、月約200ドル）が必要なため**予算ゼロと両立しない**。本計画では「自動投稿（無料枠）＋
  手動返信」から始め、月次収益がAPI費を超えた時点で完全自動化に移行する段階案を採る。
- bags系（Solana launchpad型）トークン発行は、日本居住者には資金決済法上のグレーゾーン・
  税務（雑所得）・コミュニティ反発の3重リスクがある。**禁止はしないが、発行ゲート条件
  （付録C）を満たすまで凍結**を強く推奨する。

## 1. 現状診断（Diagnostic Phase）

### 1.1 問題のMECE整理

| 領域 | 現状 | 問題 |
|---|---|---|
| **製品（Product）** | 64属性 / 相性マトリクス / 診断クイズ / 強度測定 / i18n / テスト180件 / LP・デモ動画あり | ほぼ問題なし。むしろ過剰投資。残る穴は「インストールせず30秒で体験できる導線」の弱さのみ |
| **流通・認知（Distribution）** | Star 0 / Fork 0。X・Zenn・Reddit等での発信実績なし。GitHubリポジトリ説明文が旧仕様（25属性）のまま | **最大のボトルネック。** 存在を知る経路が事実上ゼロ |
| **コミュニティ（Community）** | 外部コントリビューター0 / Discussions無効 / Issue 0 | 認知の下流なので現時点では症状。ただし受け皿（good first issue等）も未整備 |
| **収益（Monetization）** | 収益源ゼロ。Sponsors未設定 | 認知・コミュニティの下流。今やるべきは「将来の収益化を毀損しない設計」のみ |

### 1.2 根本原因分析

**5 Why（「ユーザーがいない」起点）:**

1. なぜユーザーがいない？ → 存在が知られていない
2. なぜ知られていない？ → 発信チャネルが1つも稼働していない（公開して1週間）
3. なぜ発信していない？ → 開発者の時間が100%製品開発に向いていた
4. なぜ製品に偏った？ → 「良いものを作れば広まる」前提（OSSで最も多い失敗パターン）
5. なぜその前提に？ → 配布をプロセス化した経験がなく、配布の成熟度がLv0（§1.4）

**Fishbone（要因系統図・要旨）:** 人（個人・週5-10h・マーケ経験少）/ プロセス（配布の定常
プロセスなし）/ 製品（体験までの距離が長い: pip install前提）/ 市場（「AIキャラ」分野は
AITuber・キャラbot文化で需要は実在するが、競合の自作プロンプトで代替されがち）。

**Pareto:** 成果（ユーザー獲得）に効く上位20%の活動＝「体験可能なデモ＋継続的発信」。
現在この2つへの投下がほぼ0%。

### 1.3 制約条件の特定（TOC）

```
[制約] 認知（discovery） ← システム全体のスループットを規定
   ↓ 従属させる
製品開発: 新機能開発を凍結し、メンテ＋「デモ可能性」改善のみに限定
   ↓ 制約を徹底活用する
週5-10hのうち 6-7h を発信・デモ・コミュニティに再配分（従来比ほぼ逆転）
```

制約が「認知」から「コミュニティ受け皿」に移動したサイン（Issue/質問が週3件超）が出たら
リソース配分を再調整する（§5）。

### 1.4 成熟度評価

| 軸 | レベル | 根拠 |
|---|---|---|
| エンジニアリング成熟度 | **4/5（定量管理）** | スキーマ検証・テスト180件・ruff・CHANGELOG・設計文書が揃う |
| 配布・マーケ成熟度 | **0/5（未着手）** | 発信チャネル・計測・定常プロセスすべて無し |
| コミュニティ成熟度 | **0.5/5** | CONTRIBUTING はあるが流入経路・受け皿なし |
| 収益成熟度 | **0/5** | 収益源・価格設計・法務確認すべて無し |

**含意: 上げるべきは下3つであり、上1つではない。**

### 3.4 トークン発行についての正直な評価（重要）

bags 系 launchpad（Solana）での発行自体は技術的・金銭的には容易（数千円以下）だが:

1. **法務リスク（日本居住者）**: 自己発行トークンの販売・換金は資金決済法上の暗号資産交換業
   登録要否のグレーゾーンに踏み込む。launchpad 経由で「直接販売しない」形でも、発行者として
   の継続的な換金・手数料収入には税務（雑所得・総合課税で最大55%）と規制解釈リスクが残る。
   **発行前に暗号資産案件の経験がある弁護士への相談（5-15万円）を必須とする。**
2. **評判リスク（こちらが本命）**: 日本のキャラ文化・同人文化圏は「金儲け臭」への忌避感が
   極めて強い。コミュニティ形成**前**のトークン発行は「最初から売り抜け目的のOSS」という
   ラベルを貼られ、回復不能。逆にコミュニティ形成**後**なら「ファンの参加証」として成立余地がある。
3. **経済合理性**: launchpad型トークンの大多数は流動性が数日で枯渇する。買い手＝ファンが
   存在しない現時点での発行は、収益期待値がほぼゼロのままリスクだけ先払いする行為。

**結論: 「やらない」ではなく「順序を守る」。** 6ヶ月目に付録Cのゲートで再判断。
それまでの収益化は Sponsors（リスクゼロ・OSS文化と整合）で立ち上げる。

### 4.2 ガバナンス（1人プロジェクト用に軽量化）

- **週次30分セルフレビュー（金曜固定）**: 付録Bテンプレで数字を記録 → 翌週の3タスクだけ決める。
  会議体は持たない。代わりに**すべて公開Issueで意思決定を残す**（build in public — それ自体が広報になる）。
- **月次レトロ（60分）**: KPIツリー確認、Impact/Effort再配置、止める施策を必ず1つ決める
  （週5-10hでは「やめる意思決定」が生命線）。
- **緊急時（炎上・凍結）**: ①自動投稿Actionsを即disable ②状況を固定ポストで人間として説明
  ③24時間は反論しない。この3手順を事前に PINNED_RUNBOOK.md 化しておく。

### 4.3 チェンジマネジメント（人間心理・組織行動）

対象は外部コミュニティと**あなた自身**の2つ:

- **燃え尽き対策（最大の人的リスク）**: 毎日投稿はbotに完全委譲し、人間の定常義務は
  「週30分レビュー＋週1の人格総選挙設定」だけに絞る。3週間成果ゼロでも継続できる設計が、
  個人OSSグロースの成否を分ける（効果は通常6-8週遅行する）。
- **キャラ文化圏の権利感情**: v0.x の実在キャラデータ廃止判断は正しい。発信では一貫して
  「特定キャラの再現ではなく、属性の組合せ」というフレーミングを守る。診断結果カードにも
  固有名詞を出さない（既存の固有名詞ガードをシェア導線にも適用）。
- **OSS層の「宣伝アレルギー」**: botの投稿は宣伝3:有用ネタ7（属性紹介・プロンプト技法・
  診断ネタ）。「売り込み」ではなく「面白いキャラがいる」状態を目指す。
- **トークンへの先回り**: 検討段階から公開Issueで議論する。隠して進めて発覚するのが最悪パターン。

## 5. 測定と継続的改善

### 5.1 リード指標 / ラグ指標

| 階層 | 指標 | 種別 | 目安頻度 |
|---|---|---|---|
| 活動（自分が制御可能） | 投稿数/週、記事数/月、コラボ打診数/週 | リード | 週次 |
| 反応 | プロフィール遷移率、リポジトリviews、クイズ開始数 | リード | 週次 |
| 転換 | クイズ完了数、シェア率、Star転換率（views→star） | 中間 | 週次 |
| 成果 | Stars、フォロワー、pip DL、コントリビューター数、収益 | ラグ | 月次 |

**運用原則: 週次ではリード指標だけを見て手を打ち、ラグ指標で一喜一憂しない。**

### 5.2 レビューサイクルと意思決定フロー（OODA）

- **Observe**: 金曜に計測JSONとX analytics を確認（30分の前半）
- **Orient**: 「今週いちばん反応が良かった投稿/導線は何か」を1行で言語化
- **Decide**: 翌週の3タスクと「増やすネタ/やめるネタ」を各1つ決定
- **Act**: bot のネタ帳（リポジトリ内 YAML）を更新 → 自動投稿に反映

意思決定ルール（事前コミット）:
- フォロワー500 or 収益>API費 → M2（完全自動返信）解禁
- 質問・Issueが週3件超 → リソースの2hをコミュニティ対応へ移譲（制約の移動）
- 6ヶ月時点で付録C全項目✓ → トークン発行の具体検討開始 / 1つでも✗ → 12ヶ月時点まで凍結
- 施策は6週間反応ゼロで撤退（サンクコスト切り）
